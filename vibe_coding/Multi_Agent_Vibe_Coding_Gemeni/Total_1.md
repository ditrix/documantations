Вот итог нашей сессии: краткое и структурированное руководство по мультиагентному вайб\-кодингу на стеке **React \+ TypeScript \+ Next.js**.

## ---

**Краткое руководство по мультиагентному вайб\-кодингу (TS/React)**

### **Фаза 0: Подготовка среды (The Rules)**

Вайб-кодинг начинается не с кода, а с настройки «личности» агентов в папке .cursor/rules/.

* **architect.mdc**: Описывает структуру папок (например, App Router) и стандарты типов (TypeScript interfaces).  
* **coder.mdc**: Определяет логику Server Actions, Zod-валидацию и работу с БД (Prisma).  
* **ui\_specialist.mdc**: Задает стандарты верстки (Tailwind CSS, Radix UI) и принципы интерактивности.  
* **qa\_agent.mdc**: Инструктирует ИИ проверять типы (type-check) и писать тесты (Vitest/Playwright).

### ---

**Шаг 1: Создание Блога (MVP)**

**Цель**: Развернуть базу и базовый UI.

1. **Промпт в Composer (Ctrl+I)**:"Создай Fullstack блог. **Architect**: спроектируй типы Post и Comment. **Coder**: реализуй выборку данных из SQLite. **UI Specialist**: создай адаптивную ленту постов на Tailwind. Используй types/index.ts как единственный источник правды."  
2. **Действие**: Агенты создают файлы параллельно. Архитектор задает интерфейс, Кодер пишет запрос, UI-специалист верстает компонент, опираясь на эти типы.  
3. **Вайб-чек**: Запустите npm run dev и проверьте, «дышит» ли приложение.

### ---

**Шаг 2: Расширение Блога (Auth \+ Edit)**

**Цель**: Внедрить сложную логику и проверку прав.

1. **Обновление Workflow**: Создайте или обновите workflow.md (план задач), чтобы агенты не дублировали работу друг друга.  
2. **Промпт в Composer**:"Внедряем авторизацию и правку комментов.  
   **Architect**: добавь authorId в схему.  
   **Coder**: напиши Server Action для updateComment с проверкой сессии.  
   **UI Specialist**: добавь режим инлайн-редактирования в компонент комментария.  
   **QA**: напиши тест на запрет редактирования чужого комментария."  
3. **Итерация**: Если UI-специалист ошибся в типах пропсов, QA-агент (или сам TS-сервис Cursor) подсветит ошибку, и Кодер ее исправит.

### ---

**Шаг 3: Развертывание и Контроль (CI/CD)**

Вайб-кодинг считается завершенным только после прохождения «Автоматического сита».

1. **Локальная проверка**:  
   Bash  
   npm run type-check  \# Главный судья в TS-проекте  
   npm run test        \# Проверка логики

2. **Deployment**:  
   * Агент-DevOps создает .github/workflows/deploy.yml.  
   * При каждом пуше GitHub проверяет, не сломал ли очередной «вайб» типизацию всего проекта.

### ---

**Резюме методологии**

1. **Типы прежде всего**: В React/TS проекте Архитектор всегда идет первым.  
2. **Изоляция ролей**: Не просите одного агента делать всё — разделяйте задачи по .mdc файлам.  
3. **Доверие, но проверка**: Используйте qa\_agent для запуска автоматических проверок после каждой крупной волны генерации.
